[![npm](https://img.shields.io/npm/v/@vk0/agent-claim-mcp)](https://www.npmjs.com/package/@vk0/agent-claim-mcp)
[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](./LICENSE)

**Языки:** [English](./README.md) · [日本語](./README.ja.md) · [简体中文](./README.zh-CN.md) · Русский · [Español](./README.es.md)

Используйте Agent Claim MCP, когда несколько кодинг-агентов делят один worktree и вам нужен крошечный локальный примитив координации перед правками, а не полноценный оркестрационный фреймворк. Он даёт агентам ровно одну работу: claim'ить пути, обнаруживать коллизии и release'ить владение, чтобы параллельная работа перестала затаптывать одни и те же файлы.

Agent Claim MCP — это local-first MCP-сервер для Claude Code, Cursor, Cline и других MCP-клиентов, которым нужна координация владения файлами без очередей, планировщиков и самодельных конвенций в AGENTS.md. Текущая поверхность релиза сосредоточена только на трёх ограниченных действиях: claim нормализованных путей, проверка того, кто ими владеет, и release по пути или по claim id — с явным репортингом конфликтов между разными сессиями.

## Why / When to use

Выбирайте этот сервер, когда в вашем workflow уже есть маршрутизация задач, но всё ещё нужен простой lock-подобный примитив в момент правок:

- два и более агентов могут параллельно трогать один и тот же репозиторий
- вы хотите обнаруживать коллизии до правок файлов, а не после merge-конфликтов
- вы хотите, чтобы claim'ы автоматически истекали по TTL
- вы хотите узкий примитив, который другой агент поймёт по одному только name + description
- вам **не** нужен встроенный движок правил, раннер очередей или оркестрационная платформа

## Limitations

Этот сервер намеренно узкий и local-first. Устанавливайте его, когда ваши агенты делят один и тот же вид файловой системы, а не когда вам нужна распределённая координация.

- Claim'ы живут в локальном ledger'е на диске, поэтому они координируют только те сессии, которые могут читать и писать один и тот же worktree.
- Это не хостируемый lock-сервис, не очередь и не планировщик, и он сам по себе не реплицирует claim'ы между машинами.
- Истечение по TTL — это защитная уборка, а не замена владению задачами на более высоком уровне или человеческому ревью.
- Нормализация путей уменьшает расхождения в форме путей, но агентам всё равно нужно указывать на один и тот же корень репозитория или `cwd`, чтобы claim'ы совпадали.
- Если вашему workflow нужна межхостовая координация, центральная аудит-политика или обязательные согласования, соедините этот сервер с отдельным оркестрационным слоем вместо того, чтобы растягивать claim ledger за пределы его области применения.

Реальный мультиагентный прогон перед публикацией описан в [DOGFOOD.md](./DOGFOOD.md), артефакты-доказательства — в [docs/dogfood-report.md](./docs/dogfood-report.md).

## Tools

### `claim_files`

Создаёт или обновляет локальную запись file-claim, чтобы параллельные агенты видели владение до правок и не затаптывали одни и те же пути в worktree во время активных задач. Ответы о конфликтах явные, поэтому пересекающиеся claim'ы между разными сессиями не остаются неявными или существующими только в памяти.

**Input**

```json
{
  "agentId": "coder-a",
  "taskId": "task-123",
  "paths": ["src/foo.ts", "src/bar.ts"],
  "ttlSeconds": 3600,
  "note": "working on parser cleanup",
  "cwd": "/repo"
}
```

**Output**

```json
{
  "ok": true,
  "claimed": ["/repo/src/bar.ts", "/repo/src/foo.ts"],
  "conflicts": [],
  "ledgerVersion": 1,
  "claimedUntil": "2026-05-03T16:00:00.000Z"
}
```

Если любой из запрошенных путей уже заклеймлен другим активным владельцем, tool возвращает `ok: false` и ничего не записывает.

### `release_claim`

Удаляет существующий claim, которым владеет текущий агент, чтобы завершённая, приостановленная или переназначенная работа перестала блокировать другим агентам безопасные правки тех же путей.

**Input**

```json
{
  "agentId": "coder-a",
  "paths": ["src/foo.ts"],
  "cwd": "/repo"
}
```

**Output**

```json
{
  "ok": true,
  "released": ["/repo/src/foo.ts"],
  "missing": [],
  "ledgerVersion": 1
}
```

Только текущий владелец может release'ить активный claim — независимо от того, целитесь вы по `claimId` или по нормализованным `paths`, — поэтому уборка работает после реальных мультиагентных прогонов, а варианты формы путей разрешаются консистентно.

### `whose_claim`

Читает локальный ledger и объясняет, свободен ли путь к файлу или сейчас заклеймлен, включая метаданные о владельце, задаче, note и сроке истечения — для безопасной координации между разными сессиями, делящими один worktree.

**Input**

```json
{
  "paths": ["src/foo.ts", "src/bar.ts"],
  "cwd": "/repo",
  "includeExpired": false
}
```

**Output**

```json
{
  "results": [
    {
      "path": "/repo/src/bar.ts",
      "claimed": false
    },
    {
      "path": "/repo/src/foo.ts",
      "claimed": true,
      "ownerAgentId": "coder-a",
      "taskId": "task-123",
      "note": "working on parser cleanup",
      "expiresAt": "2026-05-03T16:00:00.000Z",
      "claimId": "2a08b70c-4203-44a2-b833-31592472de1e"
    }
  ],
  "ledgerVersion": 1
}
```

## Troubleshooting

### `claim_files` возвращает `ok: false`

Это означает, что как минимум один запрошенный путь уже заклеймлен другим активным владельцем, поэтому ничего нового записано не было. Проверьте массив `conflicts` — там точные пересекающиеся нормализованные пути и текущий `ownerAgentId`, — затем осмотрите те же пути через `whose_claim` перед повторной попыткой.

```json
{
  "tool": "whose_claim",
  "arguments": {
    "paths": ["src/foo.ts", "src/new.ts"],
    "cwd": "/repo"
  }
}
```

Если конфликтует один путь, смешанный запрос всё равно остаётся all-or-nothing. Разделите работу или дождитесь, пока текущий владелец release'ит конфликтующий путь.

### `release_claim` с claim id возвращает `missing: ["..."]`

Этот claim id не был активен в локальном ledger'е на момент release. Возможно, он уже release'нут, истёк или никогда не существовал на этой машине. Это не считается частичным успехом.

### `release_claim` с paths возвращает `released: []`

Если `ok` всё ещё `true`, но `released` пустой, запрошенные нормализованные пути не совпали ни с одним активным claim'ом, которым владеет вызывающий. Массив `missing` подскажет, какие нормализованные пути не были сопоставлены.

Используйте `whose_claim` первым, когда не уверены, свободен ли путь, принадлежит ли он другому агенту или уже исчез из ledger'а.

### Release чужого claim'а

Только текущий владелец может release'ить активный claim. Если другим агентом путь всё ещё занят, осмотрите его через `whose_claim`, скоординируйтесь с владельцем или дождитесь истечения TTL.

## Comparison

| Tool | Лучше всего в | Чем Agent Claim MCP отличается |
| --- | --- | --- |
| **Agent Claim MCP** | Один узкий примитив координации для общих worktree | Он отвечает только на: кто владеет этим путём, могу ли я его заклеймить и могу ли release'ить? |
| **madebyaris/agent-orchestration** | Более широкие паттерны оркестрации, правила, маршрутизация, очереди и координация в стиле AGENTS.md | Agent Claim MCP намеренно делает гораздо меньше. Он не бандлит очереди, поведение планировщика или политику репозитория. Это самый маленький локальный примитив, который можно скомпоновать в другой workflow. |

**Выбирайте Agent Claim MCP**, когда у вас уже есть собственный слой задач и нужна только координация владения путями.

**Выбирайте более широкий оркестрационный проект**, когда вы хотите, чтобы один пакет определял правила агентов, рабочие очереди, политику координации и более высокоуровневый поток исполнения.

## FAQ

<details>
<summary><strong>Когда агенту выбирать это вместо git branch или worktree на каждого агента?</strong></summary>

Выбирайте claim'ы, когда несколько агентов намеренно делят один чекаут и один вид файловой системы — потому что они переиспользуют одни и те же `node_modules`, артефакты сборки или запущенный dev-сервер, или потому что слой задач уже делит работу по файлам, а не по веткам. Branch-per-agent и worktree-per-agent предотвращают коллизии копированием состояния и откладыванием конфликтов до момента merge; claim'ы предотвращают коллизии декларацией владения путями до первой правки — без дополнительных чекаутов и без шага merge. Если ваш workflow уже изолирует каждого агента в собственной ветке или worktree, этот сервер может вам вообще не понадобиться.
</details>

<details>
<summary><strong>Чем claim отличается от настоящего lock'а или изоляции через git worktree?</strong></summary>

Claim — это рекомендательная (advisory) запись о владении, а не принудительная файловая блокировка. Ничто физически не мешает misbehaving-процессу писать в заклеймленный путь; принуждение здесь кооперативное, и это ровно тот контракт, который MCP-агенты способны соблюдать: вызвать `claim_files` перед правками, уважать `ok: false`, release'ить по завершении. Изоляция через worktree сильнее, но тяжелее — отдельные чекауты, дублированное состояние и конфликты, которые всплывают только на merge. Эти подходы компонуются: используйте worktree для долгоживущих параллельных потоков, а claim'ы — внутри любого одного worktree, который трогает больше одного агента.
</details>

<details>
<summary><strong>Что на самом деле делают TTL и что происходит, когда claim истекает?</strong></summary>

Каждый claim несёт TTL (`ttlSeconds`, по умолчанию 3600), и истёкшие claim'ы автоматически вычищаются при следующем чтении или записи ledger'а. Это значит, что упавшая, убитая или заброшенная сессия никогда не сможет заблокировать путь навсегда — худший случай ограничен одним окном TTL. Истечение — это защитная уборка, а не передача задачи: агент, который всё ещё активно работает, должен обновлять собственный claim (переклеймить те же пути под тем же `agentId`), а не запрашивать очень длинный TTL заранее.
</details>

<details>
<summary><strong>Что именно происходит при конфликте?</strong></summary>

`claim_files` работает по принципу all-or-nothing. Если любой запрошенный путь уже принадлежит другому активному claim'у, tool возвращает `ok: false`, ничего не записывает — включая неконфликтующие пути из того же запроса — и перечисляет каждый конфликтующий нормализованный путь с его текущим `ownerAgentId` и `expiresAt` в массиве `conflicts`. Здесь нет очередей, нет ретраев и нет принудительного захвата. Что делать дальше, решает вызывающий агент: разделить пакет, осмотреть владельца через `whose_claim`, скоординироваться через собственный слой задач или дождаться release либо истечения TTL.
</details>

<details>
<summary><strong>Работает ли это между разными MCP-клиентами одновременно, например Claude Code плюс Cursor?</strong></summary>

Да — пока все клиенты работают на одной машине от имени одного пользователя. Каждый MCP-клиент запускает свой собственный процесс сервера, но все процессы сходятся к одному ledger'у на диске через атомарные записи temp-file-plus-rename, поэтому claim, сделанный из сессии Claude Code, сразу виден сессии Cursor или Cline. Единственное, что агенты должны держать консистентным, — это форма путей: передавайте `cwd` или абсолютные пути, разрешающиеся в один и тот же корень репозитория, чтобы нормализация совпадала между клиентами.
</details>

<details>
<summary><strong>Где хранится состояние и уходит ли что-нибудь с моей машины?</strong></summary>

Всё состояние живёт в одном локальном JSON ledger'е по пути `~/.agent-claim-mcp/ledger.json`. Нет демона, нет фонового процесса, нет сетевого слушателя, нет телеметрии и нет облачной синхронизации — сервер читает и пишет этот файл только в момент вызова tool'а. Это же и граница продукта: claim'ы координируют сессии, которые делят один вид файловой системы, а межхостовая координация явно вне области применения.
</details>

<details>
<summary><strong>Могу ли я release'ить или увести claim, принадлежащий другому агенту?</strong></summary>

Нет. `release_claim` удаляет только активные claim'ы, принадлежащие вызывающему `agentId`, — независимо от того, нацелены вы по `claimId` или по нормализованным путям. Если путь всё ещё принадлежит другому агенту, ваши варианты: осмотреть его через `whose_claim`, скоординироваться с владельцем через ваш слой задач или дождаться истечения TTL. Это делает примитив предсказуемым: владение меняется только явным release или истечением, и никогда — тихим захватом.
</details>

## How it works

- claim'ы хранятся в локальном JSON ledger'е по пути `~/.agent-claim-mcp/ledger.json`
- пути нормализуются в абсолютные, поэтому разные агенты не могут увернуться от коллизий трюками с относительными путями или вариантами формы путей
- пересекающиеся claim'ы между разными сессиями разрешаются через один и тот же ledger, а не через координацию в памяти
- истёкшие claim'ы автоматически вычищаются при чтениях и записях
- записи используют семантику temp-file plus rename для атомарных обновлений

## Publish prerequisites

Ожидаемый путь публикации — релизный job в GitHub Actions, который запускается после push тега версии. Чтобы безлюдный шаг npm publish прошёл успешно, workflow должен получить непустой `NODE_AUTH_TOKEN` из секрета репозитория, настроенного для публикации в npm.

Если `NODE_AUTH_TOKEN` отсутствует или пуст в CI, первая попытка npm publish падает с `ENEEDAUTH`, даже когда сам пакет готов. Перед повторным запуском релиза убедитесь, что секрет GitHub-репозитория существует и прокинут в publish-job, вместо того чтобы пытаться перезапускать шаги публикации локально.

Для локального пути доказательства перед тегированием прогоните короткий smoke-чеклист в [`docs/smoke-proof.md`](./docs/smoke-proof.md).

Для первой внешней проверки принятия после публикации используйте runbook мейнтейнера в [`docs/official-registry-validation-runbook.md`](./docs/official-registry-validation-runbook.md) и короткий шаблон записи в [`docs/official-registry-validation-checklist.md`](./docs/official-registry-validation-checklist.md).
